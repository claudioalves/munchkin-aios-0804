# Story 1.4 — Anonymous Auth flow

**Epic:** EP-01 — Setup & Infraestrutura
**Story ID:** EP01-S04
**Status:** Draft
**Points:** 2
**Branch:** `story/1.4-anonymous-auth`

## Contexto

O Munchkin Level Tracker não exige criação de conta — o usuário simplesmente abre o app e começa a jogar. Isso é possível graças à autenticação anônima do Supabase, que cria um `user_id` único por dispositivo na primeira visita e o reutiliza nas subsequentes. Esse `user_id` é a âncora do RLS: todas as políticas de segurança do banco filtram dados por `auth.uid()`, que equivale ao `user_id` da sessão anônima.

Esta story implementa a função `ensureAnonymousSession()` no pacote shared e integra o `userId` ao Zustand store, garantindo que ele persista no `localStorage`. A partir desta story, qualquer chamada ao Supabase já terá o contexto de autenticação correto, habilitando as políticas RLS criadas na Story 1.3.

## Acceptance Criteria

- [ ] AC1: `packages/shared/src/services/auth.ts` exporta `ensureAnonymousSession()` que retorna o `User` do Supabase
- [ ] AC2: Na primeira abertura do app web, `ensureAnonymousSession()` cria uma sessão anônima — verificável no Supabase Dashboard em Authentication → Users
- [ ] AC3: Nas aberturas subsequentes, `ensureAnonymousSession()` recupera a sessão existente (sem criar novo usuário)

- [ ] AC7: `npm run typecheck` e `npm run lint` passam sem erros nos arquivos criados/modificados

- [ ] T2: Criar `packages/shared/src/services/auth.ts` com a função `ensureAnonymousSession()`:
  ```typescript
  import type { User } from '@supabase/supabase-js';
  // Recebe o supabase client como parâmetro (injeção de dependência)
  export async function ensureAnonymousSession(supabase): Promise<User> {
    const { data: { session } } = await supabase.auth.getSession();
    if (session?.user) return session.user;
    const { data, error } = await supabase.auth.signInAnonymously();
    if (error) throw new Error(`Failed to create anonymous session: ${error.message}`);
    return data.user!;
  }
  ```

## Notas Técnicas

**Estrutura do `gameStore.ts` (base mínima para esta story):**
```typescript
import { create } from 'zustand';
import { persist } from 'zustand/middleware';

interface GameStoreState {
  userId: string | null;
  setUserId: (id: string) => void;
}

export const useGameStore = create<GameStoreState>()(
  persist(
    (set) => ({
      userId: null,
      setUserId: (id) => set({ userId: id }),
    }),
    {
      name: 'munchkin-store', // chave no localStorage
      partialize: (state) => ({ userId: state.userId }),
    }
  )
);
```

**Padrão de injeção do supabase client em `auth.ts`:**
A função recebe o cliente como parâmetro em vez de importá-lo diretamente. Isso mantém o `packages/shared` agnóstico à implementação do cliente (web usa Vite env vars, mobile usa Expo env vars).

```typescript
// apps/web/src/main.tsx
import { supabase } from '@/lib/supabase';
import { ensureAnonymousSession } from '@munchkin/shared';
import { useGameStore } from '@munchkin/shared';

const user = await ensureAnonymousSession(supabase);
useGameStore.getState().setUserId(user.id);
```

**Por que persistir apenas `userId` e não a sessão inteira:**
O token JWT da sessão é gerenciado pelo Supabase client internamente (em `localStorage` sob chave separada). O `userId` no store serve apenas para referência nas queries e para componentes que precisam saber "quem sou eu". Não duplicar o gerenciamento de token.
