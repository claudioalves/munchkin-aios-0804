# Story 9.9 — Fix: isolamento de dados por usuário (jogadores e partida)

**Epic:** EP-09 — Melhorias de Experiência de Jogo
**Story ID:** EP09-S09
**Status:** Done
**Points:** 3
**Branch:** `master`

## Contexto

Dois usuários logados em contas distintas compartilhavam a mesma lista de jogadores e a mesma partida ativa. O root cause era duplo:

1. **Serviços sem filtro `owner_id`**: `getActiveGame` e `getPlayers` não filtravam por `owner_id` na query. A policy RLS `games_select_active_public` (migration 002, criada para o modo espectador) permite que qualquer usuário autenticado leia todos os jogos com `status = 'active'`. Sem filtro na query, `getActiveGame` retornava o jogo do outro usuário.

2. **Zustand persist sem troca de usuário**: O store `munchkin-store` no `localStorage` persiste `activeGame` e `gamePlayers`. Ao fazer logout, `handleLogout` chamava apenas `setUserId(null)` — sem limpar os dados do jogo. O próximo login (outro usuário no mesmo dispositivo) herdava os dados do jogo anterior do localStorage.

## Root Cause Analysis

```
Usuário A loga → store: { userId: A, activeGame: {...A}, gamePlayers: [...A] }
Usuário A faz logout → handleLogout chama setUserId(null) apenas
  → store: { userId: null, activeGame: {...A}, gamePlayers: [...A] } ← VAZA!

Usuário B loga → checkSession → setUserId("B")
  → activeGame/gamePlayers do Usuário A AINDA NO STORE
  → getActiveGame sem owner_id filter → RLS games_select_active_public
    → retorna jogo do Usuário A se B não tiver jogo ativo
  → getPlayers sem owner_id filter → RLS players_select_in_active_game
    → retorna jogadores do Usuário A se estiverem em jogo ativo
```

## Fix Aplicado

### 1. `packages/shared/src/services/games.ts` — `getActiveGame`

Adicionado `supabase.auth.getUser()` + `.eq('owner_id', user.id)` na query:

```typescript
const { data: { user } } = await supabase.auth.getUser();
if (!user) return null;
// ...
.eq('status', 'active')
.eq('owner_id', user.id)   // ← novo filtro
.maybeSingle()
```

### 2. `packages/shared/src/services/players.ts` — `getPlayers`

Adicionado filtro explícito `owner_id`:

```typescript
const { data: { user } } = await supabase.auth.getUser();
if (!user) return [];
// ...
.eq('owner_id', user.id)   // ← novo filtro
.order('created_at', { ascending: true })
```

### 3. `packages/shared/src/store/gameStore.ts` — `setUserId`

`setUserId` agora limpa automaticamente os dados do jogo quando o userId muda:

```typescript
setUserId: (id) =>
  set((state) => {
    if (id !== state.userId) {
      return { userId: id, activeGame: null, gamePlayers: [], sortMode: 'custom', lastSavedAt: null };
    }
    return { userId: id };
  }),
```

Garante que qualquer troca de conta (logout → login, ou login direto de outro usuário) limpa o estado automaticamente.

### 4. `apps/web/src/pages/Home.tsx` — `handleLogout`

`handleLogout` agora chama `clearGame()` antes de `setUserId(null)`:

```typescript
clearGame();
setUserId(null);
void navigate('/auth');
```

Garante limpeza explícita mesmo que `setUserId` não detecte a diferença (e.g. race condition).

## Notas de Segurança

A policy RLS `games_select_active_public` (migration 002) foi mantida pois é necessária para o modo espectador: quem tem o link pode visualizar a partida via `getGameById`. A isolação é reforçada no nível de query (`owner_id` filter) e no nível do store (limpeza na troca de usuário). O acesso direto à API do Supabase sem o filtro de aplicação ainda exporia partidas ativas — aceitável para este tipo de app (tracker de jogo de mesa).
