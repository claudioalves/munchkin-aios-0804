# Story 5.1 — Zustand store com persist middleware

**Epic:** EP-05 — Persistência & Salvamento
**Story ID:** EP05-S01
**Status:** Draft
**Points:** 3
**Branch:** `story/5.1-zustand-persist`

## Contexto

O Zustand store com persist middleware já existe (implementado na Story 1.4). Esta story verifica que o estado persiste corretamente ao reload, adiciona o campo `lastSavedAt` ao store para uso na restauração (Story 5.4), e garante que os dados necessários estão no `partialize`.

## Acceptance Criteria

- [ ] AC1: Reload da página mantém `activeGame`, `gamePlayers`, `userId` e `sortMode` do localStorage
- [ ] AC2: Campo `lastSavedAt: string | null` adicionado ao store — atualizado a cada mudança de nível
- [ ] AC3: `setLastSavedAt(timestamp)` action disponível no store
- [ ] AC4: `lastSavedAt` incluído no `partialize` (persiste no localStorage)
- [ ] AC5: `npm run lint` e `npm run typecheck` passam sem erros

## Tasks de Implementação

- [ ] T1: Adicionar `lastSavedAt: string | null` e `setLastSavedAt(timestamp: string)` ao `GameStoreState` em `packages/shared/src/store/gameStore.ts`
- [ ] T2: Incluir `lastSavedAt` no `partialize`
- [ ] T3: Executar `npm run lint` e `npm run typecheck`

## Notas Técnicas

**Adição ao store:**
```typescript
lastSavedAt: string | null;
setLastSavedAt: (timestamp: string) => void;
```

**Inicialização:**
```typescript
lastSavedAt: null,
setLastSavedAt: (timestamp) => set({ lastSavedAt: timestamp }),
```

`lastSavedAt` será chamado pelo `useLevelUpdate` após cada save bem-sucedido no Supabase, e pelo `useAutoSave` após snapshots.

## Dependências

Story 1.4 (store base), Story 4.2 (useLevelUpdate).
